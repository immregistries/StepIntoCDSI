This work concerns Clinical Decision Support for Immunizations (CDSi) forecasting as defined by the CDC CDSi logic specification. CDSi describes a deterministic process that evaluates a patient’s immunization history, determines dose validity and series status across possible vaccination series, and produces forecast dates for the next indicated dose. The processing model is inherently series-centric and iterative: multiple relevant patient series may be evaluated independently using rule tables and interval logic, after which a best series result is selected to represent the patient’s overall recommendation. Because orchestration of this evaluation and forecasting loop governs the final outcome, small routing or state-handling errors can invalidate large portions of expected results.

Step Into CDSi (Step) is a student-developed reference implementation intended to reproduce the CDC logic as faithfully as possible in executable software. The system loads standardized NIST test cases, computes evaluation and forecast results, and compares them to expected outcomes to measure conformance. The implementation is incomplete and currently contains defects—particularly in orchestration logic—so the primary goal is systematic convergence toward near-complete test pass rates through iterative debugging and correction.

Internally, the architecture mirrors the structure of the CDC specification. Each major chapter or logical phase is represented by a dedicated Logic Step class within the core logic package, allowing the execution flow to follow the same conceptual progression as the written guidance. A central DataModel object carries all state for a single evaluation session; each logic step reads from and writes to this shared model and determines which subsequent logic step should execute next. Within individual steps, Logic Tables replicate the decision tables defined in the CDSi specification, preserving rule behavior as directly as possible from the source guidance. As a result, most defect resolution occurs within a specific logic step or its associated tables rather than across the entire system.

Execution is orchestrated through two primary entry points. The ForecastServlet performs the full chained evaluation and forecasting workflow to generate final CDS results. In contrast, the StepServlet exposes a step-through interface that allows a human user to advance logic step by step while inspecting intermediate state and reasoning. This stepping capability serves both as a debugging mechanism and as an educational tool for understanding how the CDSi specification operates in practice.

The objective of this effort is iterative conformance improvement. By running focused test scenarios, examining detailed execution traces, identifying divergence from the specification, applying targeted fixes, and verifying results through regression testing, the implementation is expected to converge toward full CDSi correctness. Within this workflow, AI is used as a diagnostic and analytical assistant—helping interpret traces, hypothesize root causes, and suggest minimal corrective changes—while final validation remains grounded in deterministic testing against the official CDSi expectations.
